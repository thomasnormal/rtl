# Engineering Log

## 2026-03-30

- The right upgrade path for the weakest RISC-V bundles was not to invent hidden infrastructure; it
  was to surface the normalized public contract directly in the checked-in task library. `aplic_idc`
  and `imsic_interrupt_file` already had manifest-backed public interfaces in the task-store path,
  but until those `interface/` contracts and solver-facing README/dv docs were checked in beside the
  source markdown, the bundles still looked like raw spec fragments rather than tasks.
- The `riscv_hardware_specs` review also needed a sharper distinction between active tasks and raw
  source packs. `external_debug` and `advanced_interrupt_architecture` are useful retained source
  material, but scoring them as if they were benchmark tasks hid the real state of the active pack,
  which is currently two marginal public tasks that are blocked mainly by missing oracles.
- For the weak `uart_slave` AVIP task, the missing piece was public usability, not more hidden
  oracle detail. The local package enums and named UVM tests already pinned down baud rates, parity,
  framing, break behavior, payload widths, and oversampling modes; the fix was to summarize that
  into a checked-in `protocol_summary.md` so solvers no longer have to reverse-engineer the public
  contract from package files and test names.
- The right way to repair older converted AVIP docs was not another blind translation pass; it was
  to keep the cleaned markdown, recover the original rendered PDF pages, and recrop the missing
  figures directly from those page images. That let the AHB document be rewritten into real
  chapter markdown while restoring figure coverage for labels like `Fig 3.8`, `Fig 8.12`, and
  `Fig 9.15` that had been left dangling as prose only.
- GitHub markdown rendering does not surface image alt text as a visible caption. For AVIP docs,
  important figure labels need explicit markdown caption lines under the image, otherwise a tighter
  crop can make the figure text disappear entirely from the rendered page.
- The earlier AVIP repair pass was strong enough to reject literal full-page renders, but not
  strong enough to reject page-like crops with excess whitespace. `AhbAvipArchitectureDocument`
  Figure 2.1 was the concrete miss: the right fix was a tighter crop plus chapter-level markdown
  cleanup around the figure so GitHub renders it like documentation instead of a page dump.

- `adc_ctrl` needed two extra opaque public-ABI widths in the OpenTitan self-contained projector: `ast_pkg::adc_ast_req_t` is 3 bits and `ast_pkg::adc_ast_rsp_t` is 11 bits in the current AST package. `lc_ctrl_pkg::lc_tx_t` is also a common 4-bit opaque control type that should stay in the shared width map for future IPs.
- The first end-to-end `rv_timer` generator run exposed a trainer integration gap rather than a task issue: `validate_generator_episode()` only dispatched legacy sim and cocotb oracles, so OpenTitan `opentitan_dvsim` tasks died with “task rv_timer has no oracle” even though the hidden metadata was present. The right fix is to route those tasks through the OpenTitan validator, not special-case them in the batch runner.
- `rv_timer` is currently the smallest OpenTitan task in the local pack by public interface size (9 ports), which makes it the right first end-to-end target. The `gpt-5.4` generator produced a self-contained submission that passed its own local `xrun` smoke/self-check benches, but the hidden repo-native `rv_timer_random` UVM test still timed out in `cip_base_vseq`, so the local smoke was too weak and the candidate is not yet OpenTitan-correct.
- The first `rv_timer` rerun uncovered a generic bug in our OpenTitan public register projection: IPs with `alert_list` need a synthesized `ALERT_TEST` register at offset `0x0` before the explicit HJSON register list. Without that, the public regs package shifted documented offsets like `CTRL` down by one word and misled both the generator and verifier.
- After fixing the projected register map and strengthening the generator instructions around documented CSR side effects, the second `rv_timer` candidate materially improved: it now implements `ALERT_TEST`, compare-write interrupt clear, invalid-access error responses, and the documented reset/register defaults, and it passes a much stronger local `xrun` smoke bench. The hidden OpenTitan oracle still fails in `rv_timer_random`, so the remaining gap is no longer the obvious CSR map bugs from the first run.
- The verifier agrees with the oracle label on that improved `rv_timer` candidate, but not for the same reason. Its `xrun`-based public checks all passed, and it returned `bad` only because the public task ABI still does not expose a usable way to stimulate the documented `BUS.INTEGRITY` / fatal-alert requirement. That means the current public TL helper package is still too lossy for full verification.
- A likely remaining root cause of the repo-native UVM timeout is that the generated candidate only drives a subset of the packed TL response bits (`d_valid`, `d_opcode`, `d_data`, `d_error`, `a_ready`) and ignores response metadata such as `d_source` and `d_size`. Our current public TL helper package also only exposes that reduced subset. Even if that is not the only bug, the next OpenTitan task-format improvement should be a fuller public TL ABI with helpers for all semantically relevant request/response fields and, where possible, an explicit public integrity-fault stimulus contract.
- The AVIP PDF converter hang was in the trainer wrapper, not the model prompt. `run_converter_opencode()` used `Popen(..., stdout=PIPE, stderr=PIPE)` and, once output looked complete, sent `SIGTERM` only to the top-level `opencode` process before calling `communicate()`. If a child process still inherited those pipe file descriptors, `communicate()` could block forever waiting for EOF even though the markdown and manifest had already been written.
- The right fix was process-group management, not more timeout heuristics. The converter now starts OpenCode in a new session and terminates or kills the whole process group on settle/timeout, which matches what we actually mean by “stop the agent run.” A regression test now covers the exact failure mode with a child process that keeps stdio open after the parent exits, and a real `gpt-5.4-mini` UART conversion returned cleanly in about 65 seconds after the change.
- The shutdown fix also held on a real nontrivial AVIP conversion: the 51-page `spi_avip_specification.pdf` run completed with full page-read coverage, wrote section markdown plus figures and `manifest.json`, and returned control to the wrapper cleanly after about 590 seconds instead of leaving the parent shell stuck.
- The separate `top_module.txt` turned out not to buy much. The simpler public format is `task/task.json` with a single `top_module` field, plus `task/spec/interface/*.sv` as the canonical HDL boundary. That keeps one public metadata file without reintroducing duplicated interface structure into JSON.
- Once the public interface SV became canonical, the right micro-arch structural check was still to consume those public files directly. The bind-check helper now reads `task/task.json` only for the DUT name and uses the generated `*_public_if.sv` for the actual boundary, which keeps the public contract small without inventing another top-module file.
- The two remaining OpenTitan pack holdouts, `dma` and `sysrst_ctrl`, did not need new oracle framework code. Both already had upstream smoke tests (`dma_generic_smoke` and `sysrst_ctrl_smoke`) and dropped cleanly into the existing `opentitan_dvsim` adapter once the manifest copied their RTL into `oracle/golden_rtl/`.
- Gold selftests now pass for all nine curated OpenTitan IP tasks, so the pack is fully reference-backed at the hidden-oracle level. The next maturity gap is no longer “missing runnable oracle,” but “how much of the upstream DV can be reused against arbitrary generated candidates.”
- `sysrst_ctrl` is the clearest example of that next gap: its upstream DV collateral still references internal DUT hierarchy in places like the coverage interface hookup and `SYSRST_CTRL_DUT_TOP` macro. Gold selftest is fine because the golden RTL matches that structure, but a truly implementation-agnostic candidate flow will likely need an explicit public `micro_arch/` ABI before the adapted oracle is robust.
- The right public OpenTitan task shape is now stable: every curated IP publishes `task/spec/interface/*.sv` plus `task/spec/micro_arch/*.sv`. The micro-architecture ABI is no longer a UART / RV timer special case; it is part of the default medium-tier contract for the whole OpenTitan pack.
- A candidate wrapper and a gold selftest wrapper do not want the same structure. Candidate validation benefits from a top-level alias of the candidate’s named micro-architecture interface so the public bind/checker files can attach generically. Gold selftests, on the other hand, must preserve the native DUT hierarchy for upstream SVA and CSR assertions. The clean split is: alias wrapper only for candidate validation, direct top-module stub injection for gold selftests.
- Once the wrapper exposed a top-level alias, the public `*_micro_arch_bind.sv` files became reusable inside the hidden oracle instead of needing per-IP bind-path rewrites. That is the scalable direction: keep the public ABI stable and move only the repo-native adaptation into a thin hidden overlay when a specific DV file insists on an old path.
- UART still carried a stale hidden `compat/` ABI in its local DV core file even after the public task moved to `micro_arch/`. The right fix was not to resurrect `compat` publicly, but to add thin hidden shims in `dv/compat/` that forward the old repo-native include names to the public `uart_micro_arch_*` ABI.
- After that split, the OpenTitan gold-selftest sweep is green again with public `micro_arch/` enabled across the entire 9-IP pack. The authoritative summary is `runs/opentitan_oracle_micro_arch_v3/summary.json`.
- A public completeness audit of the 9-IP OpenTitan pack showed that the main spec/docs were already mirrored, but we were still dropping useful **public DV collateral** such as `dv/*_sim_cfg.hjson` and `dv/doc/*.svg`. Those files are now mirrored into the public task bundles as well, and a regression test compares each materialized task against the relevant upstream public files under `README.md`, `doc/`, `data/`, and `dv/`.
- The audit result is now: the OpenTitan tasks are document-complete relative to the selected public upstream collateral. The remaining uncertainty is task difficulty, not missing public docs. `dma` is still the least friendly task because of boundary complexity, and `sysrst_ctrl` still inherits an upstream doc gap because there is no separate programmer guide to mirror.
- Mirroring raw upstream public docs was the wrong optimization for the agent-facing task surface. Even when the docs are public, repo-specific clutter like OpenTitan badges, `hw/ip/...` paths, `dvsim` commands, CIP references, raw sim cfg files, and testplan HJSON distracts the generator from the actual design problem.
- The right public projection for OpenTitan-derived tasks is a **normalized task-facing spec layer**: keep the relevant behavior, interfaces, registers, and micro-architecture ABI, but strip or rewrite upstream project-specific material. The raw upstream docs can still exist in the task library or hidden source bundle for traceability; they should not be the primary public task surface seen by the agent.
- The AVIP doc converter needed a second structural gate beyond figure-crop validation: some promoted outputs were still organized as page dumps (`full.md`, `spec.md`, raw `page-*.png`, or `chunk_###/...`) rather than chapter files. The converter now rejects those layouts, and the AVIP batch script no longer chunks large PDFs by page range unless explicitly opted into with `--allow-page-chunks`.
- Older AVIP conversions still need spot repair even after the converter is hardened. The AHB architecture doc had a full-page page-render masquerading as `figure-009.png`, and its front matter was a low-quality OCR dump; fixing it by hand made it clear that navigational pages like TOCs, figure lists, and table lists should be reconstructed into markdown tables instead of preserved as line-broken page text.
- The remaining "older AVIP" cleanup was best handled as a deterministic in-place repair instead of another long full-PDF reconversion. The key repair pattern was: detect stale full-page rendered PNGs in active task-library docs, split pages that had been reused for multiple figures into distinct cropped assets, and add a regression audit that forbids both page-sized figure assets and duplicate image references in active AVIP markdown.
- The `rv_timer` generator run showed that “write a smoke test and inspect a waveform” is still too weak for medium-tier tasks with a public `micro_arch/` ABI. The candidate passed its happy-path smoke and wave review but silently treated `intr_state_0` as a gated interrupt output instead of a raw pending source state. The prompt now forces the generator to read `micro_arch/README.md`, turn each exported ABI signal into an explicit requirement, add executable checks for every exported signal, and write at least one negative test when a micro-arch signal could diverge from a masked/gated/latching public output.
- The staged OpenCode environment was breaking user-installed Python CLI tools like `vcdcat` by rewriting `HOME` to a workspace-local directory. `vcdcat` itself lives in `~/.local/bin` and imports `vcdvcd` from the Python user site, so once `HOME` changed, `/usr/bin/python3` started looking under the staged `.home/.local/...` instead of the real user base. The clean fix is to keep the isolated workspace `HOME`/XDG paths but preserve `PYTHONUSERBASE` from the original shell (or default it back to the original `~/.local`) so user-site Python tools keep working inside staged runs.
- Staged OpenCode runs were also relying entirely on the parent process environment for provider credentials. In this shell, `OPENAI_API_KEY` was not exported even though the repo `.env` contained it, so explicit `--model openai/...` runs failed immediately with `ProviderModelNotFoundError`. The runtime now merges missing env vars from the repo `.env` into the staged process environment without copying `.env` into the public workspace.
- For AVIP figure extraction, "use loose crops" was not strong enough. The AHB implementation chapter showed the practical failure mode: tight crops silently clipped code blocks and flowchart edges. The better default is to stage gridded page copies for coordinate selection, crop the final asset from the raw page image, and bias both the prompt and the regression tests toward slightly undercropped figures rather than aggressive trimming.
- A repo-wide figure audit caught three distinct classes of broken documentation assets outside the original AHB cleanup: active markdown still pointing at staging-only `pages/page-*.png` files, active docs carrying page-sized “figure” renders, and missing promoted assets where the markdown named a figure file that had never been created. The durable fix is a single task-library doc-asset regression that checks AVIP and RISC-V docs together, not more one-off repairs.
- A useful task-quality rubric has to match the repository's actual task format, not generic dataset language. In this repo the strongest predictors of a good task are: whether the public spec is self-contained, whether the DUT boundary is explicit in machine-checkable files, whether the oracle passes gold selftests and actually checks meaningful behavior, and whether the task resists shortcut solutions. Difficulty matters, but it should not outweigh oracle quality or public-spec clarity.
- The self-containment bar needs to be stricter than "mostly materialized." The desired end state is zero dependency on the original upstream repo for either the public task or the oracle. If oracle utility code is genuinely needed, the right move is to copy or rewrite it into a task-local or trainer-owned form, not to preserve a hidden assumption that some upstream checkout still exists.
- "Spec quality" and "documentation quality" are not the same thing. A task may contain the right information but still be poor for agents if that information is only available through a raw PDF, bad OCR, broken chapter splits, or low-quality figures. The rubric should score both separately: one axis for whether the intended behavior is publicly available at all, and another for whether the model can read it easily and reliably.
- For agent-run acceptance, "the model eventually wrote RTL and evidence" is not enough; the run is not operationally complete until `result/result.json` exists. Fresh `aplic_idc` generator reruns showed a real close-out failure mode where the agent built RTL, benches, waveforms, and successful `xrun` evidence, then kept iterating instead of emitting the summary bundle. The public generator/verifier prompts and staged `TASK.md` instructions now state explicitly that once `result/result.json` matches the evidence, the agent should stop immediately and avoid extra cleanup/polish loops.
- Prompt wording alone was still too weak for `gpt-5.4-mini`. The stronger control-plane contract is now: stage a provisional `result/result.json` stub with `status: "in_progress"`, teach the runtime to ignore non-terminal stubs, and tell the agent to update that existing file immediately after the first requirement pass instead of treating the summary bundle as a late end-of-run artifact. Even with that change, fresh `aplic_idc` generator reruns still showed the mini model preferring more evidence work over updating the stub, so the remaining blocker is model behavior rather than missing workspace affordances.
- The OpenCode wrapper was also treating a placeholder `result/result.json` as terminal as soon as the file stabilized, even when the agent had only written `{"status":"in_progress", ...}`. That can silently cut off longer verifier episodes before they finish their benches. The wrapper now parses the JSON payload and only terminates early once the result file is both stable and terminal; `in_progress`, `pending`, and `running` are explicitly nonterminal.
- The next close-out failure mode was verifier-specific: after a directed `xrun` smoke test had already produced concrete spec failures, the agent still kept searching for more evidence instead of writing `verdict: bad`. The fix is to make that decision rule explicit in both the verifier prompt and staged `TASK.md`: one reproducible executable failure on a critical requirement is already enough to close out a negative verdict.
- After fixing staged provider env loading from the repo `.env`, explicit `openai/gpt-5.4` verifier runs now start and reach real `xrun` evidence instead of dying with provider lookup errors. A fresh `rv_timer` rerun got through elaboration, generated assertions/bind files plus a self-checking smoke bench, and produced a passing public `xrun` smoke log without mutating the candidate. The remaining operational gap is close-out: the agent still left `result/result.json` in `status: "in_progress"` even after the smoke evidence existed, so the next verifier-control improvement should focus on promoting completed evidence into a terminal verdict more reliably.
- The project needed one durable place to see “where things stand” without re-reading manifests, test logs, and run directories by hand. The right first scoreboard is derived, not curated: read `configs/datasets.json`, count local `data/task_store/<dataset>/<task>/task.json` materializations, and summarize the latest generator/verifier `summary.json` artifacts under `runs/` into one markdown table.
- Picking the "latest" run by file mtime alone was too noisy once verifier shards and one-off smoke runs accumulated under `runs/`. A more useful scoreboard rank is "broadest completed coverage first, newest only as a tiebreaker", and merged verifier summaries must tolerate `examples_requested: null` so they still appear as the current dataset-level result.
- The current `aplic_idc` generator miss exposed a real SystemVerilog gotcha rather than a task-specific semantic gap: a zero-argument helper function called from a continuous assign can leave derived combinational outputs stale under Xcelium because the hidden dependencies are not part of the assign sensitivity. The durable control-plane fix is to teach the generator to compute those values in `always_comb` or to pass every dependency as an explicit function argument, not to special-case APLIC.
- The follow-on `aplic_idc` verifier miss showed the symmetric testbench gotcha: after a testbench changes combinational DUT inputs, sampling outputs in the same procedural instant is too strict for native SV scheduling. A candidate that settles correctly one delta later can be misclassified as bad if the bench does not wait at least `#0`, `#1step`, or a tiny local delay such as `#1ps` before checking derived outputs.
- The same operational failure mode that motivated verifier timeout-closeout exists on the generator side too: a run can produce real RTL and evidence but never promote the stubbed `result/result.json` out of `in_progress`. The generic `run_opencode()` closeout path was already sufficient; the missing piece was setting a generator-specific timeout closeout prompt in `prepare_generator_episode()` so long generator runs can still terminate with a truthful best-effort summary.
- Generator timeout closeout needs the same audit trail as verifier closeout. The preserved first-pass OpenCode stdout/stderr logs are not just for negative verification verdicts; they also capture partial compile errors, unfinished smoke checks, and self-reported implementation gaps from long generator runs. The generator closeout prompt now explicitly reads those transcript files before writing its final `result/result.json`, so a timed-out generation run can still leave a truthful terminal summary instead of discarding its most informative evidence.
- The next generator-control bug only showed up on real medium-tier runs: a generator can exit cleanly with return code 0 while leaving `result/result.json` in `status: "in_progress"`. The wrapper was only forcing closeout on timeout, so `rv_timer` and `aon_timer` ended up archived with real artifacts on disk but no terminal generator summary. The correct rule is broader: if an agent process exits and there is still no terminal result payload, launch the same short closeout pass before returning control.
- The clean control-plane fix for verifier close-out is a second short OpenCode pass in the same workspace, not a heuristic wrapper verdict. When a verifier run hits its time budget, the runtime now terminates the long run and launches a dedicated timeout-closeout prompt that is only allowed to read existing evidence and update `result/result.json` with a mandatory best-effort `good`/`bad` verdict. A fake-OpenCode regression now covers the exact failure mode: first pass writes `in_progress` evidence and hangs, second pass writes the terminal verdict, and the wrapper returns a `forced_closeout` result instead of timing out.
- The first real `rv_timer` verifier rerun with forced closeout exposed one more control-plane gap: the terminal verdict was correct, but the raw simulator transcript in `xrun_smoke.log` could still be truncated if the long first pass died mid-tool-run. The runtime now persists the interrupted first-pass OpenCode stdout/stderr into `result/evidence/timeout_closeout_pass1_stdout.log` and `...stderr.log` before launching closeout, and the closeout prompt explicitly tells the verifier to use those files when they exist. That keeps the full interrupted transcript available for audit even when a tool-owned log file is incomplete.
- The OpenTitan hidden overlay path needed a clean separation between candidate-mode and gold-mode DV assets. The right structure is `repo_overlay/common/`, `repo_overlay/public/`, and `repo_overlay/gold/`, with common files staged into both worktrees and mode-specific files only into the corresponding candidate or gold run. That avoids teaching the public task about upstream hierarchy while still letting gold selftests keep the native OpenTitan structure.
- Using hardlinks while building OpenTitan oracle worktrees was a real correctness bug, not just an optimization risk. A candidate-mode overlay could mutate the same inode that a gold selftest later reused, which meant one oracle run could silently affect another. The repo copy path now uses normal file copies for oracle worktrees so overlays are isolated the same way commercial verification flows expect separate compile work areas to be isolated.
- `aon_timer` is now a useful medium-tier oracle task in the intended sense: gold selftest still passes, and candidate validation no longer dies on hidden `tb.dut.u_reg...` hierarchy assumptions. After replacing the candidate-mode bind, smoke sequence, base test, and scoreboard environment with public-ABI-safe overlays, the candidate now builds and reaches the real OpenTitan `aon_timer_smoke` run phase, where it fails with a meaningful UVM timeout instead of a hidden hierarchy error.
- The latest end-to-end `aon_timer` sample also makes the current bottleneck explicit. The generator can read the curated public task, write a substantial draft RTL, and mirror the public packages, but it still times out before creating any self-checking bench, xrun compile, or waveform evidence. The verifier, meanwhile, gets through public elaboration but not through executable behavioral checks before its closeout pass. So the next medium-tier improvement target is no longer task clarity or oracle compatibility; it is pushing both agents to reach executable evidence earlier in the run.
- Flat 600s/900s agent timeouts were too small once the tasks moved from RTLLM-scale problems to medium-tier public-spec problems. The control plane now resolves timeouts from task tier instead of dataset name: `small` stays at 900s, but `medium` defaults to 2400s for the generator, 1800s for the verifier, and 1800s for the oracle. That keeps short tasks cheap while giving OpenTitan-scale runs enough wall-clock budget to reach executable evidence before timeout closeout.
