# Engineering Log

## 2026-03-27

- The repo already had the right public-task abstraction for spec-only corpora: a public `spec/` tree with no oracle metadata. The missing piece was not a new storage model, but checking in one real curated pack and proving it through the existing task-store path.
- Spec-only public packs do not need special ingestion logic if the source artifact can live under a normal task-library directory. Copying the official PDFs into `task_library/riscv_hardware_specs/<task>/` kept the implementation simple and let the same curated-pack path materialize public-only tasks without hidden assets.
- The first spec-only corpus is now `riscv_hardware_specs`, using official RISC-V International PDFs for External Debug v0.13.2 and Advanced Interrupt Architecture v1.0. The task-store smoke run and regression tests now cover a public-only curated pack with no oracle and no private source bundle.
- Freezing a local git checkout into a shared source bundle is not enough if the registry still keeps the live dirty checkout around. The shared-source registry now prunes the stale `git_checkout` entry when snapshotting, so OpenTitan tasks point at exactly one clean hidden source root.
- Depending on OpenTitan's upstream `reggen` at runtime is brittle in this environment because the current upstream utility code assumes Python 3.10 union-type syntax while this repo runs under Python 3.9. The public OpenTitan register helper packages are now generated by a direct HJSON parser in this repo instead of importing upstream Python tooling.
- The self-contained OpenTitan public ABI needed more than opaque typedef projection. The task store now emits task-local TL-UL helper packages and register/field constant packages next to the generated public interface SV, so agents can drive bus traffic and CSR programming from machine-checkable SV rather than prose.
- The hidden OpenTitan source bundle should be shared, but the task materialization must still run against the frozen snapshot root, not the live checkout root. The task-store now uses the frozen bundle for interface projection, hidden oracle assets, and source metadata, while recording the original checkout path separately for provenance.
- Repo-native OpenTitan gold selftests cannot assume the source DV environment is untouched. The local UART DV tree already expects `dut.u_candidate.u_uart_compat_if`, so the gold path now wraps the copied golden RTL in a native-port `u_candidate` wrapper, injects a compat instance, and extends the staged sim core to compile `dv/compat/*.sv`.
- Hidden repo overlays are candidate-adaptation assets, not universal patches. The oracle prep now distinguishes gold/public compat modes, stages real compat collateral for gold and candidate validation, and applies hidden repo overlays only in the modes that actually need the adapted bench.
- The UART medium-tier oracle is now in the shape we wanted: the copied golden RTL passes the repo-native `uart_smoke` flow again, and the three curated UART mutants all fail for meaningful reasons (`tx_enable_tied_low` hits top-level and DUT assertions, `tx_done_interrupt_disabled` fails the interrupt scoreboard, and `rx_data_zeroed` fails the smoke flow).
- The same shared-bundle plus copied-golden-RTL template now works for more than UART. `i2c_host_smoke` and `spi_host_smoke` both pass against copied golden RTL, which is enough to treat `uart`, `i2c`, and `spi_host` as the first real medium-tier OpenTitan oracle-backed tasks.
- The PDF converter should treat the multimodal agent as the source of truth, not `pdftotext` or OCR helpers. The converter prompt and staged `TASK.md` now require a page-image-first workflow, explicit figure extraction into `output/figures/`, markdown references of the form `![Figure ...](figures/figure-042.png)`, and allow manual crops with PIL when a visual should stay as an image.
- A quick AVIP figure-cropping experiment on rendered PDF pages showed that explicit image-space crop boxes `(x1, y1, x2, y2)` work reliably for isolated diagrams and boxed figures when the coordinates are in page-pixel space with a top-left origin.
- The same crop-box method is less reliable as an agent contract for mixed-content pages because the hard part is not the crop operation itself, but deciding whether the crop should include captions, section labels, or surrounding explanatory text. A slightly tighter SPI crop still looked visually fine but silently dropped the figure caption, which is exactly the kind of brittle failure that matters for spec fidelity.
- Dense tables like the APB signal table are a bad fit for figure extraction by default; they should stay as markdown text unless the multimodal agent explicitly decides the layout cannot be preserved textually.
- The most robust default policy for multimodal PDF conversion is: transcribe the page first, keep figures as text when feasible, otherwise reference a whole-page or loose figure crop, and reserve tight region crops for pages with multiple unrelated visuals.
- A real GPT-5.4 smoke run on the one-page UART AVIP architecture PDF produced the right output shape: a section markdown file (`01_architecture.md`), a matching manifest, and an extracted figure asset referenced from the markdown. The figure crop was correct and the prose covered the visible labels and connectivity exhaustively enough for this simple page.
- The remaining weakness in that UART smoke run is not missing content but style: the model tends to write mechanically exhaustive prose about diagram layout ("upper portion of the page", "blank white space below it") that is faithful but lower-signal than the actual architectural content. Future prompt tuning should push it toward exhaustive technical content while de-emphasizing irrelevant page-layout narration.
- The converter control plane now pre-renders PDF pages into `input/pages/` before the agent starts and validates the OpenCode trace to make sure every `page-*.png` was opened through the `read` tool at least once. The early-termination path now waits for both a complete markdown manifest and complete page-read coverage before killing a still-running agent.
- The page-read hook is intentionally narrow: it ignores bash/PIL access used for figure cropping and only treats `read(...)` hits on the pre-rendered page images as the completeness sanity signal.
- A real smoke run with a built-in OpenCode multimodal model (`opencode/mimo-v2-omni-free`) passed the new gate on the one-page UART PDF: the agent produced markdown output and the workspace-local OpenCode log recorded a `read` event for the pre-rendered page image.
- A separate smoke attempt with `openai/gpt-5.4` from the isolated workspace failed before conversion because that OpenCode session reported `Model not found: openai/gpt-5.4`. That is a provider/model wiring issue in the isolated OpenCode environment, not a failure of the new page-read validation logic.
- The public OpenTitan task boundary needed a stricter split between external interface and deep-DV ABI. `task/spec/interface/` is now the canonical standalone DUT boundary, while `task/spec/micro_arch/` is the optional explicit verification ABI for deeper oracle reuse.
- The `micro_arch/` surface is now validated at ingest time instead of being treated as loose collateral. A public microarchitecture contract must include `README.md`, exactly one `*_micro_arch_if.sv`, a `dut` modport, and at least one consumer modport such as `tb` or `mon`.
- `rv_timer` is now the second concrete public microarchitecture profile after UART. Its public ABI explicitly exports timer state and integrity/alert observation points (`ctrl_active_0`, `intr_state_0`, `timer_val_0`, `compare_val_0_0`, `tl_intg_error_pulse`, `fatal_alert_pulse`) so deeper verification can be expressed without hidden hierarchy peeks.
- The first `rv_timer` verifier/oracle mismatch showed that the public TL helper package was too lossy. The generated public TL-UL package now exposes request and response metadata such as source, size, param, user, sink, and integrity fields, not just payload data/error bits.
- Even comments can weaken the public boundary. The generated public SV interface packages no longer mention hidden native OpenTitan types like `tlul_pkg::tl_h2d_t` or `prim_alert_pkg::alert_rx_t`; those mappings stay only in hidden task metadata.
- The agent-visible contract now says explicitly what we had only been implying: the staged `task/` directory is the complete public problem statement. Generators and verifiers should not assume any access to upstream repo code, hidden packages, or hidden hierarchy outside that staged workspace.
- A public microarchitecture contract is useful only if it fails fast and concretely. A native `xrun` elaboration of the improved `rv_timer` candidate plus the public `micro_arch` bind/checker files now fails immediately on the missing `u_rv_timer_micro_arch_if` instance and exported signals, which is exactly the explicit structural signal we wanted instead of a delayed opaque UVM timeout.
- That bind-check path is now generic rather than handwritten. The repo has a reusable helper that reads `task/task.json` plus `task/spec/micro_arch/`, generates a tiny top that instantiates the DUT and bind module, and lets `xrun -elaborate` answer “does this candidate satisfy the public microarchitecture ABI at all?” before heavier verification starts.
- "Gold-pass" is not the right universal oracle invariant. The right split is reference-backed oracles, reference-free executable oracles, and public-only tasks. OpenTitan fits the first bucket; many other task families do not.

- Gold selftests now pass for all nine curated OpenTitan IP tasks, so the pack is fully reference-backed at the hidden-oracle level. The next maturity gap is no longer “missing runnable oracle,” but “how much of the upstream DV can be reused against arbitrary generated candidates.”
- `sysrst_ctrl` is the clearest example of that next gap: its upstream DV collateral still references internal DUT hierarchy in places like the coverage interface hookup and `SYSRST_CTRL_DUT_TOP` macro. Gold selftest is fine because the golden RTL matches that structure, but a truly implementation-agnostic candidate flow will likely need an explicit public `micro_arch/` ABI before the adapted oracle is robust.
