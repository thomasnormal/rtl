# Engineering Log

## 2026-03-25

- The workspace started empty, so the first useful deliverable is a minimal but runnable scaffold, not partial integrations into an existing codebase.
- The first bootstrap needs to be verifiable. I added tests for dataset manifests, reward logic, Tinker group advantages, and episode-budget behavior before wiring in more ambitious training code.
- RTLLM is the cleanest anchor-seed source because it exposes spec, testbench, and verified RTL in one task directory.
- VerilogEval is still important even if the long-term system is agentic; it gives a maintained public harness and a useful held-out benchmark.
- AssertEval is the right side-dataset for generated-evidence skills because it is directly about formal assertions rather than plain code generation.
- AssertEval has a minor source mismatch: the live AssertLLM repo README says 20 designs, while the OpenLLM-RTL paper summary says 18. The manifest notes the discrepancy instead of pretending it does not exist.
- OpenTitan and Ibex are strategically correct but tactically later. They are too heavy for phase 0 and should not block the small-task verifier environment.
- The local Python is `3.9.25`, so packaging metadata must stay honest even if the code would also run on newer interpreters.
- `python -m rtl_verifier.cli ...` was initially silent because the module lacked an entry guard. The package entrypoint is now wired for both `python -m rtl_verifier` and the installed `rtl-verifier` script.
- The real RTLLM repository layout is nested and uses per-design `verified_*.v` filenames. The earlier flat `verified_verilog.v` assumption was wrong and is now fixed in discovery plus normalization.
- The current RTLLM GitHub repo exposes `main` and `v1.1` as separate branches, so the fetch flow should materialize both instead of treating one checkout as both datasets.
- This machine has `xrun`, `iverilog`, `vvp`, `yosys`, and `sby`. The local oracle path should prefer `xrun`, with `iverilog` only as a fallback.
- RTLLM gold selftests need a top-module adaptation step because the shipped gold RTL often defines `verified_<name>` while the testbench instantiates `<name>`. The simulation runner now rewrites only the first module declaration for that gold-selftest path.
- The first architecture was still too agent-visible: it exposed normalization and oracle execution as a normal package/CLI surface. The cleaner design is a trainer-owned task store plus staged OpenCode workspaces that contain only public task inputs.
- “Full bash” and “hidden oracle” are only compatible if the bash access is scoped to the staged workspace. The repo now makes that boundary explicit with OpenCode workspaces and `external_directory: deny`.
- The first real RTLLM oracle smoke test failed for a good reason: the trainer runtime was invoking `xrun` from a changed working directory while still passing relative source paths. `load_stored_task` and the oracle plan builders now resolve task, work, and candidate paths to absolute paths before simulation.
- The first end-to-end live run now works on a real dataset task. `RTLLM-v1.1` was materialized into `data/task_store/rtllm_v1_1` with 29 tasks, and an OpenCode generator run on `adder_8bit` produced RTL that passed the hidden `xrun` oracle.
- Long baseline runs need resume support. The first 29-task batch died partway through, and the clean fix was to make the batch runner reuse existing `batch_record.json` files and rerun only unfinished task directories.
- The trustworthy batch artifact is the per-task `result/batch_record.json`, not a half-finished stdout dump. For large runs, regenerating `summary.json` from finished records is safer than relying on a blocked CLI process to print a giant terminal summary.
- The first full RTLLM v1.1 baseline with `openai/gpt-5-mini` under OpenCode is now complete: 14 of 29 tasks passed the hidden oracle, 14 produced candidate RTL that failed oracle validation, and 1 (`RAM`) exited without writing `submission/candidate.sv`.
- The RTLLM oracle packaging was too weak in four specific ways: it dropped sidecar files like `reference.dat` and `test_data.dat`, it chose the first module instead of the DUT module for multi-module gold RTL, it treated some `.v` testbenches as plain Verilog even when they used SystemVerilog syntax, and it trusted success markers even when `xrun` had already logged `*E` tool errors.
- The right fix was to strengthen the hidden oracle pipeline, not to patch individual tasks. The task store now copies all hidden support files, the oracle runner stages every hidden sim input into the work directory, `xrun` runs with `-sv`, and oracle judgment reads the `xrun` log instead of only stdout/stderr.
- Two RTLLM v1.1 anchors are not salvageable as trustworthy hidden oracles: `div_16bit` has a broken published testbench, and `radix2_div` has a published gold RTL that fails its own benchmark. Those tasks are now quarantined from the anchor set by default.
- After rebuilding `data/task_store/rtllm_v1_1`, the clean anchor set is 27 tasks. All 27 gold designs now pass their own hidden oracle in `runs/gold_selftests_fixed/summary.json`.
- Revalidating the original `gpt-5-mini` batch against the repaired oracle moved the baseline from `14/29` to `16/27`. The false failures were `calendar` and `multi_booth_8bit`; the remaining failures are candidate bugs or missing candidate output, not packaging mistakes.
- The earlier `RAM` “missing candidate” failure was not a hard RTL problem and not an unclear interface. A fresh rerun of the same task produced `submission/candidate.sv` and passed the hidden oracle, so that miss was a transient OpenCode/runtime failure.
- Even when the runtime behaves, the public task contract was too weak. The generator now gets a structured `task/task.json` contract with extracted module name, inputs, outputs, parameters, and deliverable paths, and `TASK.md` explicitly says that contract is authoritative.
- The structured-spec extractor was still too brittle for RTLLM v1.1 anchor tasks. The right fix was to stop inferring those interfaces and check in a manual `configs/rtllm_v1_1_interfaces.json` manifest that covers all 29 published tasks.
- The clean anchor set remains 27 tasks because `div_16bit` and `radix2_div` are still quarantined as bad oracles, but they now also have curated public interfaces so the contract stays complete even when `include_invalid=True`.
- The first verifier smoke run exposed an integrity bug: the verifier edited the staged candidate RTL to add a timescale before judging it. The fix was twofold: stage verifier candidates read-only in the workspace, and have the verifier benchmark explicitly disqualify any episode that mutates `candidate/`.
- Per-example OpenCode failures must be recorded as verifier misses, not crash the whole benchmark. The verifier batch runner now catches timeouts/errors and writes a result record with `predicted_verdict = null` and the execution error.
- The first meaningful verifier baseline is now in `runs/verifier_benchmark/rtllm_v1_1_run01_gpt_5_mini_20260326T0100Z`: on the 27-task clean `run_01` corpus, `openai/gpt-5-mini` reached `22/27 = 81.5%` accuracy with 2 false negatives, 2 false positives, 1 timeout, and 0 candidate mutations.
- The verifier prompt was still underspecified: “write SVAs or UVM-style checks” is not enough. The prompt and skills now tell the agent to keep the DUT immutable, put assertions in separate or bind files, prefer `xrun` for native SV/SVA simulation, and use `xrun -uvm` for real UVM environments instead of treating UVM as a vague architecture pattern.
- After the repo was initialized with git, staged episode directories stopped being isolated in practice: OpenCode could walk up to `/home/thomas-ahle/rtl/.git` and treat the whole repo as the project. The launcher now passes `--dir <workspace>` explicitly and sets `GIT_CEILING_DIRECTORIES` to the workspace parent so git discovery stops at the episode boundary.
- The next isolation step is to get episode workspaces out of the repo entirely. Batch and verifier episodes now stage under `/tmp/rtl-episodes` by default and only move back into `runs/...` after the agent finishes, which removes the parent git repo from the agent’s live working tree.
- Staging under `/tmp` is not enough by itself if OpenCode still sees user-global config and plugins. The launcher now points `HOME`, `XDG_*`, `OPENCODE_CONFIG`, and `OPENCODE_CONFIG_DIR` at workspace-local paths so each episode uses only the checked-in repo prompts/skills copied into that staged workspace.
- The verifier should not frame the tool choice as only “SVA or UVM.” Cocotb is a legitimate third option when a Python reference model or scoreboard is materially cheaper than pure SV, so the prompt and `sv-verification` skill now explain when to use cocotb and show a minimal `@cocotb.test()` pattern.
- The project needs a clearer spec-data ladder, not just “more benchmarks.” The right split is public small anchors, public medium IP manuals, public large subsystem/SoC manuals, and a separate industrial-private lane for licensed standards like JEDEC DDR. Mixing those into one manifest without access metadata would be a future legal and evaluation mess.
- The first OpenTitan medium-tier pack should be curated, not auto-ingested, but that does not mean rewriting the spec. The repo now carries manually selected copies of the upstream OpenTitan docs for `uart`, `i2c`, `spi_host`, `dma`, and `sysrst_ctrl`, while the task metadata and interface manifest stay curated and manifest-driven.
- OpenTitan medium-tier tasks still do not have a runnable hidden oracle in the RTLLM sense, but the materialized task store now references a shared hidden source bundle, keyed by the local checkout path plus git commit/dirty metadata, so future oracle runners can build from the real implementation and DV collateral without copying the whole repo into every task.
- The public interface contract needed a real HDL form, not just JSON. The task store now normalizes every public interface into a canonical port list and materializes `public/spec/interface/<top>_public_if.sv` with generated `dut` and `tb` modports, so agents can read a concrete SV boundary contract instead of reconstructing one from prose or JSON.
- OpenTitan exposed a data-shape bug in our own manifests: those tasks had `inputs` and `outputs` but an empty `ports` list. Normalizing the public interface at ingest time fixed that class of inconsistency once, instead of relying on every downstream consumer to special-case the manifest format.
- The first runnable OpenTitan oracle should use a frozen hidden reference, not the live checkout by accident. The UART task now copies `hw/ip/uart/rtl` into `oracle/golden_rtl/` and carries raw `opentitan_dvsim` metadata that points at the upstream `uart_smoke` flow.
- `dvsim` root discovery is git-based, so a staged OpenTitan repo nested under `/home/thomas-ahle/rtl/...` will otherwise resolve the wrong `proj_root`. The right fix is to pass `--proj-root <staged_repo>` explicitly when launching the repo-native oracle.
- A gold selftest is necessary but not sufficient. After wiring the UART `dvsim` oracle, the copied golden RTL passed `uart_smoke`, and a one-line negative control that changed `cio_tx_en_o` from `1'b1` to `1'b0` failed on the expected top-level and DUT assertions. That gives us the first medium-tier oracle that is both runnable and discriminative.
- The first OpenTitan UART generator attempt with `gpt-5-mini` produced a syntactically plausible top-level stub, but it optimized for the public interface summary rather than the repo-native build contract. The generated `uart.sv` omitted the upstream `import uart_reg_pkg::*;`, so `NumAlerts` was undefined during the real Xcelium build, and it instantiated `compat_if` instead of the ABI-required `u_uart_compat_if`.
- For medium-tier tasks, "generate a top module matching the interface" is not enough. The generator prompt or the public compatibility artifacts need to force exact package imports, parameter typing, and named compatibility instances if we want repo-native DV to be usable as the hidden oracle.
- The UART stub was not caused by missing source material. The trace shows the generator only read `TASK.md`, `task.json`, `uart_public_if.sv`, `uart_compat_if.sv`, and `uart_compat_checker.sv`; it never opened the functional spec chapters. That means the current workspace instructions are overweighting the machine-readable interface and weak compatibility checker, and underweighting the requirement to read and implement the behavioral spec.
- For package-heavy SV tasks like OpenTitan IP, `yosys` is the wrong default sanity check. The generator instructions now prefer `xrun`/Xcelium for compile/elaboration checks and demote `yosys` to a fallback for small standalone RTL, which matches the actual oracle environment much better.
- A soft “prefer xrun” instruction was not strong enough. On the next UART rerun, the agent still chose `yosys` for the final compile check, so the generator contract now says that for OpenTitan-style tasks, package-typed ports, or compat-driven tasks, `xrun` is the required compile check and `yosys` does not satisfy that requirement.
- Requiring the generator to preserve or import upstream OpenTitan packages is the wrong abstraction boundary. The public task should be self-contained; any repo-native package adaptation belongs in the hidden oracle wrapper, not in `submission/`.
- The right OpenTitan public ABI is not “the upstream package names, but public.” The task store now projects package-typed OpenTitan ports into self-contained task-local SV support files such as `uart_public_types_pkg.sv`, while preserving the repo-native port types only in hidden task metadata for oracle use.
- That projection can be generic across the current OpenTitan pack without hand-writing per-IP wrappers: the public side aliases opaque package types to fixed-width task-local typedefs, and the hidden oracle wrapper bit-casts between those task-local types and the repo-native package types port by port.
- Gold selftests and candidate validation do not want the same compat staging behavior. The upstream UART DV flow now expects a `dv/compat` directory, but the public compat ABI is candidate-facing and stronger than the published gold top, so the oracle stages no-op compat stubs for gold selftests and the real public compat files for candidate validation.
- After the OpenTitan ABI refactor, the real UART gold selftest still passes through the repo-native `dvsim` flow. That means the public task boundary is now self-contained for agents without breaking the hidden commercial-tool oracle.
- The next UART rerun with `gpt-5.4` did read the full functional spec and wrote a real requirement checklist, and the submission bundle became properly self-contained: `submission/uart.sv` plus `submission/uart_public_types_pkg.sv`, with no `` `include`` back into `task/`.
- OpenCode itself is not yet a reliable batch primitive for medium-tier runs. On the `gpt-5.4` UART episode it wrote the candidate and local `xrun` evidence, but then failed to return control to the wrapper, so the runner had to archive the staged workspace from `/tmp/rtl-episodes` and continue oracle evaluation out of band.
- The first repo-native candidate validation failures after that rerun were still oracle-wrapper bugs, not RTL verdicts. The wrapper header parser stopped at `import uart_reg_pkg::*;`, which truncated OpenTitan-style module headers before the parameter and port lists, and then the direct bit-cast path failed on array-typed native ports like `prim_alert_pkg::alert_tx_t [NumAlerts-1:0]`.
- The right fixes were structural. The wrapper parser is now a small state machine that understands `module ... import ...; #( ... ) ( ... );`, and cast-required ports are emitted through local typedef aliases so Xcelium sees legal named cast targets instead of raw array type expressions.
- Reusing OpenTitan DV against a wrapped candidate needs a hidden repo overlay layer, not more prompt pressure. Curated task packs now support raw-oracle assets from the task-library root, OpenTitan oracles can overlay hidden `repo_overlay/` files onto the staged shared repo, and the UART task now uses that to replace only `dv/tb/tb.sv` and `dv/compat/uart_compat_bind.sv` with compatibility-aware versions.
- With those oracle repairs in place, the UART candidate from `gpt-5.4` now gets through repo-native build and elaboration and fails in the actual `uart_smoke` run phase with a real `cip_base_vseq wait timeout`, which is the first medium-tier negative result from the adapted OpenTitan oracle that is clearly about the candidate behavior rather than harness breakage.
